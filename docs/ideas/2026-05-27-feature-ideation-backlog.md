## 1. 어종 추가

**무엇:** 빌트인 어종 로스터 확장(현재 5종 → +N종).

- 코드는 이미 **레지스트리 패턴**으로 거의 공짜. `speciesRegistry.ts`에
  "GLB 추가 → import 1줄 → `SpeciesId` 유니온 1줄 → `SPECIES_REGISTRY` 항목 1개"면 끝.
  `fishAssets.loadFishPrototypes`가 자동 로드.
- **진짜 병목은 코드가 아니라 에셋**: CC0 라이선스 + **스켈레탈 swim 클립 포함 GLB** +
  머리 +X 규약 + 적절한 폴리수. 상세는 `2026-05-27-fish-model-research.md`.
- 어종 추가 시 `kind`(schooling/individual), `baseScale`, `swimSpeed`, `displayName`,
  `dialogue`(10개) 작성 필요.
- **YAGNI 경고:** "사용자가 런타임에 직접 GLB 임포트"까지 가면 보안(임의 GLB 로드)·검증·
  에셋 영속화·UI 때문에 난이도가 급상승. 1차 범위는 **빌트인 확장만** 권장.

**리스크:** 모델별 머리 방향/자세 검증 필요(GLB 노드 변환 무시 시 누운/뒤집힌 자세 — CLAUDE.md 함정).

## 2. 배경(바닥) 테마 추가

**무엇:** Aquascape 바닥/하드스케이프를 선택 가능한 테마로(모래/자갈/수초밭/암반 등).

- 현재는 절차적 모래 + 하드스케이프 단일 구성.
- 접근: (a) 절차적 변형(텍스처/색/노이즈 파라미터화) (b) GLB 하드스케이프 세트 (c) 둘 혼합.
- **최대 리스크 = 투명 셰이더 함정.** HANDOFF에 기록된 "모래 평면 먼 가장자리 하드 컷 →
  전체폭 수평선", "반투명/additive 평면이 바탕화면 위에 사각형으로 비침" 버그를 그대로 밟기 쉽다.
  새 배경은 반드시 **알파 페이드(smoothstep) 규약**을 따르고 고밝기+밝은배경 격리 캡처로 검증.
- `THREE.Fog` 금지, additive는 알파도 누적.

**리스크:** 시각 회귀가 가장 잦은 영역. 비전 eval보다 격리 캡처(고밝기) 게이트가 핵심.

## 3. AI 사용량 앰비언트 모니터  ⭐ 최우선 후보

**무엇:** 사용 중인 AI(예: Claude Code) 사용량을 수족관 상태로 앰비언트 표현
(codex 펫과 유사). 5시간 롤링 한도 잔량·토큰 사용량 등.

- **데이터 소스(현실 체크):** Claude Code는 `~/.claude/projects/.../*.jsonl` 세션 로그에
  메시지별 토큰 수를 남긴다. 5시간 롤링 윈도우 사용량도 이 로그로 추정 가능
  (ccusage류 커뮤니티 도구가 쓰는 방식). **네트워크/공식 API 없이 로컬 파일 파싱으로 가능.**
- **아키텍처:** 파일 읽기는 **반드시 main 프로세스**(renderer 보안 규칙). 파싱·집계 →
  순수 함수로 분리(TDD) → IPC로 renderer에 사용량 스냅샷 전달 → 수족관 표현.
- **표현 설계가 핵심(브레인스토밍 필요):** 후보 —
  - 잔량↓ → 물이 점점 탁해짐(틴트/베일 강도)
  - 잔량↓ → 물고기 유영 느려짐/활력 감소
  - 작은 게이지/숫자(흘긋 보기용, 기본 숨김 토글)
  - 한도 리셋 시점에 "먹이 시간" 같은 작은 이벤트
- **차별성 최상** — 위젯을 수동 장식에서 앰비언트 정보판으로 전환.

**리스크:** "정확한 5시간 잔량"은 비공식 추정 → ±오차. 로그 포맷 변경 시 깨질 수 있음(방어적 파싱).
표현이 과하면 힐링 컨셉을 해침(절제 필요).

## B. 시간대/무드 반응  ⭐ 저비용 보너스

**무엇:** 실제 시각에 따라 물 색·조명 자동 변화(아침 블루 → 한낮 밝음 → 노을 → 밤 어둡고 발광).

- 기존 **밝기/IBL/`environmentIntensity`/깊이 틴트** 파이프라인을 시각→파라미터 매핑으로 재사용.
- 순수 함수 `timeOfDay → {tint, intensity, ...}` 매핑(TDD), 렌더는 기존 setter 호출.
- 인터랙션 0, 순수 앰비언트. **싸고 무드 효과 큼.**
- 선택: 수동 오버라이드(밝기 슬라이더)와의 우선순위 정의 필요.

**리스크:** 거의 없음. 사용자 수동 밝기 설정과 자동 무드의 충돌 규칙만 정하면 됨.

## C. 물고기 성장/유대 (다마고치류)

**무엇:** 먹이 준 물고기가 시간에 따라 자라고, 이름이 붙고, 재시작해도 상태 유지.

- 장기 애착 루프. `persistence.ts`(localStorage) 인프라 이미 존재.
- 난이도 상: 개체별 영속 상태 모델(나이/크기/이름/마지막 먹이 시각), 풀링과의 정합,
  성장에 따른 scale 변화 등.

**리스크:** 상태 모델 복잡도. 힐링 위젯에 "돌봐야 하는 의무감"이 생기면 역효과(밸런싱 주의).

## E. AI 작업 상태 연동 (활발/유휴)

**무엇:** 정확한 잔량 대신 **지금 AI가 길게 작업 중인가/유휴인가**에 반응
(작업 중 → 물고기 활발·기포↑, 유휴 → 고요).

- 3번보다 **감지가 단순하고 신뢰도 높을 수 있음**(최근 로그 활동 타임스탬프만 봐도 됨).
- 3번과 결합 가능(잔량 = 무드, 활동 = 활력).

**리스크:** "작업 중" 판정 기준 정의(최근 N초 내 로그 갱신 등). 3번과 데이터 소스 공유 설계.

## D. 앰비언트 사운드

**무엇:** 잔잔한 물소리/기포음, 토글.

- 저비용. 힐링 컨셉과 어울림.
- **기본 OFF 필수** — 상주 always-on-top 위젯이라 소리가 거슬리면 즉시 삭제 대상.

**리스크:** 상주 위젯 소음 민감도. 볼륨·기본값 신중.
